Creating the Gradle Project

Let's jump right into creating that FastJ project.

Before following this tutorial...

Before following this tutorial, please make sure you've installed all the requirements for creating a FastJ project outlined here.

Create a Workspace#

Now that you have your IDE and other tools installed (if you don't, click here to get them), we can start off by giving ourselves a folder to work in. Open up your file explorer on your pc, and create a folder to work inside of.

For the sake of this project, I've named mine fastj-tutorial -- feel free to name it the same if you're not confident in what exactly you're doing.

Be organized!

Like with other computer tasks, it's best practice to stay organized while working. Putting in the effort to stay organized now instead of later makes your life much easier down the line.

Open The Terminal/Command Prompt#

Next, you'll want to open up a terminal (Linux/Unix) or a command prompt (Windows) inside of the folder you created. The instructions below are divided by the operating system -- click or tap the operating system name to change to its instructions.

  • Windows
  • macOS
  • Linux
Type cmd into File Explorer's searchbar -- this opens the command prompt and directs it to the folder you're in.
Until further notice, I don't own a macOS device and don't actually know how to do this off-hand. Instead, follow these guidelines.
Right-click anywhere inside the folder window, and select Open in Terminal.

Project Type Selection#

Gradle Project Type Selection
Select type of project to generate:  1: basic  2: application  3: library  4: Gradle pluginEnter selection (default: basic) [1..4]

This section is asking you what type of Gradle project you want to make. Gradle supports a few different default configurations of projects:

  • application - executable applications like games and software apps
  • library - literally libraries of code game-making code frameworks, UI frameworks
  • Gradle plugins - code to improve your build script experience
  • basic - an empty project with no specialization whatsoever.

Each one has its own pros and cons, and are generally designed to speed up the development process of setting up your project.

For now, we can just pick number 2 -- application. This configuration gets us the quickest to making our game. In more advanced situations, we may end up using the other templates, but for now we can stick with just this template.

Programming Language Selection#

Programming Language Selection
Select implementation language:  1: C++  2: Groovy  3: Java  4: Kotlin  5: Scala  6: SwiftEnter selection (default: Java) [1..6]

This section is much more straightorward -- it's only asking what programming language we're going to be using. As we know, this tutorial will be using Java so you can go ahead and select number 3 -- Java.

Functionality Splitting#

Functionality Splitting
Split functionality across multiple subprojects?:  1: no - only one application project  2: yes - application and library projectsEnter selection (default: no - only one application project) [1..2]

This one might be a bit confusing. However, fear not -- we're just going to stick with choice number 1 -- only one application project.

What's a subproject?

Gradle has this functionality called modules -- it's the idea of separating your one big project into multiple chunks, or subprojects. Dividing your project allows you to separate logic from implementation, which is a very good way to create reusable code. I won't go into detail because it's out of the scope of this tutorial, but this is also another way useful way of organizing your project and one step into the ways of abstraction.

Oh, and in case you're curious about using multiple subprojects you can check out Gradle's guide here.

Build Script#

Build Script
Select build script DSL:  1: Groovy  2: KotlinEnter selection (default: Groovy) [1..2]

This one's once again more straightorward, although you may not have heard of build scripts before. A build script is a set of code defining a set of rules for the build tool to follow. This gives the build tool the context it needs in order to work for your specific use cases.

I personally prefer Groovy to Kotlin (this is preference!), so we'll use number 1 -- Groovy.

Source Package Name#

Source Package Name
Project name (default: FastJ Tutorial Project):

This one's almost as simple as project name, but we'll need to go much more in-depth in order to get it right. Java packages have to follow a style guide that defines what the top package names (the one we're deciding right now) can be.

  1. The product/project name needs to be specified as the last part of the package sequence.
  2. The package sequence needs to be written in reverse domain name notation.

The project name is simple enough -- fastjtutorial will do. (Java packages don't allow spaces or dashes, so we'll just put the two words together.)

But what is reverse domain name notation, you ask. Reverse domain name notation is the way website URLs are written when you search them, but backwards.

Here's an example:

  • Domain: docs.google.com
  • Reverse Domain Name Notation: com.google.docs

That hopefully simplifies how this works.

But what do we actually put for this? Well, we need to use a domain that we own. Luckily, everyone is entitled to a specific domain thanks to Github, a site where you can store your code privately or publicly, and share it with others on the internet. As long as you have a Github account, you can (legally) use this package sequence:

io.github.yourusername.yourprojectname

That's exactly what we'll use: io.github.yourusername.fastjtutorial.

What if I don't want a Github account?

For future reference, there will likely be parts of this tutorial that involve you having a Github account, so that may be a slight hinderance to you in the long run.

However, you don't need to follow these conventions if you aren't distributing your code to others. It's another good practice -- one that might bite you in the butt if you don't use. You've been warned!
